# Drizzle Migrations

> Read this before adding a column, creating a table, writing an RLS policy, or touching anything under `packages/db/`. The general migration discipline in `_rules/common/database-migrations.md` applies on top of these Drizzle/Supabase-specific rules.

## Toolchain

| Tool | Role |
|---|---|
| `drizzle-kit generate` | Diffs your TypeScript schema against the migrations folder, writes a new SQL migration |
| `drizzle-kit migrate` | Applies pending migrations to the configured DB (use this in prod) |
| `drizzle-kit push` | Direct schema push — **dev/local only**, never prod |
| Supabase CLI (`supabase migration new`) | Owns auth/storage/realtime migrations; we own `public.*` |

## Repo layout

```
packages/db/
  schema/
    index.ts             # barrel — re-exports all tables
    users.ts             # profiles, memberships
    orgs.ts              # organizations
    catalogue.ts         # brands, categories, products, variants
    commerce.ts          # carts, quotes, orders, invoices
  migrations/
    0000_<auto>.sql      # generated by drizzle-kit
    0001_<auto>.sql
    meta/
      _journal.json      # drizzle-managed — DO NOT EDIT
      0000_snapshot.json
      0001_snapshot.json
  policies/
    quotes.sql           # RLS policies for quotes
    orders.sql           # RLS policies for orders
    products.sql         # RLS policies for products
  drizzle.config.ts
```

## Workflow: adding a table

1. **Define the table in `packages/db/schema/<file>.ts`** using `pgTable`. Include `org_id` if the data is org-scoped.
2. **Run `pnpm db:generate`** (alias for `drizzle-kit generate`). This writes `packages/db/migrations/NNNN_<auto>.sql`.
3. **Review the generated SQL.** Drizzle's diffing isn't always what you want — check for unwanted column drops, missing defaults, or wrong constraint order.
4. **Add the RLS policy file** in `packages/db/policies/<table>.sql`. The Drizzle-generated migration enables RLS via a `-- statement-breakpoint` block that you append manually, or via a follow-up migration that runs the policy SQL.
5. **Apply locally:** `pnpm db:migrate` (alias for `drizzle-kit migrate` against the local Supabase Docker DB).
6. **Test the policy** per `_rules/supabase/rls.md`.

Until step 6 passes, the migration is not "done."

## Schema patterns

### Standard table starter

```ts
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core';

export const quotes = pgTable('quotes', {
  id: uuid('id').primaryKey().defaultRandom(),
  orgId: uuid('org_id').notNull().references(() => organizations.id, { onDelete: 'restrict' }),
  status: text('status', { enum: ['requested', 'quoted', 'accepted', 'rejected', 'expired'] }).notNull().default('requested'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull().defaultNow(),
});
```

- UUIDs for primary keys (Postgres `gen_random_uuid()` via `defaultRandom()`).
- `text` with `enum:` for status fields, backed by a `CHECK` constraint Drizzle generates automatically.
- `timestamp({ withTimezone: true })` — always `timestamptz`, never plain `timestamp`.
- `onDelete: 'restrict'` for FK to org — never silently cascade away order history.

### JSONB specs

```ts
import { jsonb } from 'drizzle-orm/pg-core';

export const products = pgTable('products', {
  id: uuid('id').primaryKey().defaultRandom(),
  brandId: uuid('brand_id').notNull(),
  modelNumber: text('model_number').notNull(),
  specs: jsonb('specs').$type<ProductSpecs>().notNull().default({}),
});
```

Type the JSONB shape with `$type<T>()`. Validate writes with Zod at the service-module boundary — Postgres won't enforce shape, only Zod will. See `_rules/common/postgres-patterns.md` for JSONB indexing.

### Unique constraints and indexes

```ts
import { uniqueIndex, index } from 'drizzle-orm/pg-core';

export const products = pgTable(
  'products',
  { /* columns */ },
  (t) => ({
    brandModelUnique: uniqueIndex('products_brand_model_uq').on(t.brandId, t.modelNumber),
    specsGin: index('products_specs_gin').using('gin', t.specs),
    nameFts: index('products_name_fts').using('gin', sql`to_tsvector('english', ${t.name})`),
  }),
);
```

GIN indexes on JSONB and on `to_tsvector` are the two patterns we lean on hardest in Phase 1.

## RLS policies — separate SQL files

Drizzle's TS DSL doesn't model RLS. Keep policies as plain SQL alongside migrations:

```sql
-- packages/db/policies/quotes.sql

ALTER TABLE quotes ENABLE ROW LEVEL SECURITY;

CREATE POLICY "quotes_select_by_org_member" ON quotes FOR SELECT
  USING (public.is_org_member(org_id));

CREATE POLICY "quotes_insert_by_org_member" ON quotes FOR INSERT
  WITH CHECK (public.is_org_member(org_id));
```

Apply via either:
- Append the SQL into the Drizzle-generated migration file before applying (preferred — keeps everything in one transaction)
- Run as a follow-up migration (acceptable if the policy is added later)

Either way, the policy SQL is version-controlled in `policies/` for diffability, even if the canonical apply path is the migration.

## The "never edit an applied migration" rule

Once a migration appears in `_journal.json` and has been applied anywhere outside your laptop, it is immutable. To change schema:

- **Add a new migration** that alters the table.
- **Never** rewrite the old `NNNN_<auto>.sql` file. Doing so silently desynchronises every developer's local DB and breaks prod-vs-local diffs.

If you must fix a typo in a fresh migration (literally hours old, only on your machine):
1. Delete the migration file
2. Delete the corresponding `meta/NNNN_snapshot.json`
3. Manually undo the `_journal.json` entry
4. `supabase db reset` to wipe local
5. Re-run `pnpm db:generate`

If the migration has been merged or applied to a shared DB, do not undo — write a follow-up.

## Production migration command

Wire the prod migration into CI as `drizzle-kit migrate`, never `drizzle-kit push`:

```bash
# In CI, after build, before deploy
pnpm exec drizzle-kit migrate
```

`push` is a schema-sync command — useful for local prototyping but destructive in prod (will drop columns to match the schema). Banish it from any non-local environment.

## Seeding

Seeds go in `packages/db/seeds/<purpose>.ts`. Use the same Drizzle client + admin DB connection. Idempotent: every seed must be safe to re-run (use `ON CONFLICT DO NOTHING` or check existence first).

```ts
// packages/db/seeds/brands.ts
import { db } from '../client';
import { brands } from '../schema/catalogue';

export async function seedBrands() {
  await db.insert(brands).values([
    { name: 'Waaree', slug: 'waaree' },
    { name: 'Jinko', slug: 'jinko' },
    /* ... */
  ]).onConflictDoNothing({ target: brands.slug });
}
```

## Anti-patterns

- **Editing `_journal.json` to "fix" a migration.** Path to a corrupted DB. Use a follow-up migration instead.
- **Calling `drizzle-kit push` against any DB that isn't your local Docker Supabase.** Will silently drop columns.
- **Forgetting `pnpm db:generate` after editing `schema/*.ts`.** The schema and the migrations diverge — runtime errors with cryptic messages.
- **Putting RLS policies inline in TS files via raw SQL strings.** Loses the diff-friendly `.sql` files. Keep policies in `packages/db/policies/`.
- **Skipping the `.notNull().default(...)` on a new non-nullable column** when there's existing data. The migration will fail. Add the column nullable, backfill, then `ALTER COLUMN SET NOT NULL` in a follow-up.
- **Reusing the same migration name twice.** Drizzle numbers them automatically — don't override.
